# Decentralized KYC: Conceptual & Technical Guide

This document provides a deep dive into the concepts, architectures, and implementations of the **Decentralized KYC (Know Your Customer) Blockchain Project**. This guide is designed to help you understand the "why" and "how" of the project for your presentation.

---

## 1. Project Overview
The goal of this project is to eliminate the redundant and intrusive nature of traditional KYC processes. Instead of uploading your documents to every bank or service individually, we use blockchain and Zero-Knowledge Proofs to create a **unified, privacy-preserving identity system**.

### The Core Problem
- **Data Silos**: Each institution keeps its own copy of your data (risky).
- **Redundancy**: Users repeat KYC for every new service.
- **Privacy Loss**: You often reveal your entire ID (DOB, Address) just to prove you are "over 18".

### The Solution: "Verify Once, Share Anywhere"
Users verify their identity with a trusted authority (e.g., Government) once. After that, they can share **proofs** of their identity with other entities (Banks, Exchanges) without necessarily sharing the raw data.

---

## 2. Core Blockchain Concepts

### A. Decentralized Identity (DID)
We implement a form of **Self-Sovereign Identity (SSI)**. The user is the owner of their data. They grant and revoke access to their "Identity Profile" on the blockchain.

### B. Immutability & Transparency
The blockchain provides a "Single Source of Truth". Once a document hash is anchored, it cannot be altered without changing the hash, making forgery impossible. 

### C. Trustless Verification
The blockchain acts as a "Trust Layer". A bank doesn't need to trust the user; they trust the **Government's signature/hash** anchored on the blockchain.

---

## 3. Smart Contract Architecture

The project utilizes three main smart contracts to manage logic, security, and privacy.

### 1. `DecentralizedKYC.sol` (The Core Logic)
This is the heart of the system.
- **User Registry**: Tracks registered users and their status.
- **Entity Registry**: Manages a list of "Verified Verifiers" (Banks, etc.).
- **Access Control**: Implements `grantAccess()` and `revokeAccess()`. An entity cannot view a user's verification status unless the user explicitly allows it on-chain.
- **Verification History**: Stores an immutable log of who verified whom and when.

### 2. `MultiSigWallet.sol` (The Governance Layer)
To prevent a single "God Admin" from having too much power, we use a Multi-Signature approach for the Government role.
- **Consensus**: High-stakes actions (like registering a new Bank) require **threshold approval** (e.g., 2 out of 3 admins must sign).
- **Security**: Protects the system from a single compromised admin account.

### 3. `ZKVerifier.sol` (The Privacy Layer)
This contract is generated by **SnarkJS**. It contains the mathematical logic to verify Zero-Knowledge Proofs (zk-SNARKs). It returns a simple `true` or `false` without knowing the underlying data.

## 4. Specific Implemented Technologies

### A. Zero-Knowledge Proofs (ZKP)
We use **Circom** to write ZK circuits that allow "Selective Disclosure".
- **Implemented Case**: Proving age >= 18.
- **Private Input**: User’s actual age (stored only in the user's browser/vault).
- **Public Signal**: The boolean result (1 or 0) and the threshold (18).
- **Verification**: The smart contract verifies the proof without seeing the age.

### B. Secure ID Vault & IPFS
Documents are not stored on-chain (too expensive and public).
- **IPFS Storage**: Encrypted documents are stored on IPFS.
- **ID Vault**: The backend route `api/vault` manages these document links and their local hashes.
- **Data Integrity**: The hash of the IPFS file is what gets put on the blockchain. If even 1 pixel of the ID changes, the hash won't match.

### C. Public Ledger (Real-time Auditing)
We implemented a **Public Ledger** dashboard that uses **Socket.io** to listen for blockchain events in real-time.
- **Transparency**: Anyone can see when a KYC event happens (anonymously).
- **Auditability**: Regulators can track system activity without seeing sensitive user data.

## 5. Hash-Based Verification Flow (The "Anchor & Match" Model)

A critical feature of this project is how entities (like Banks) verify a user without requiring a central database to "hand over" the data.

### Step 1: Government "Anchoring" (Root of Trust)
When a user first joins the system, a **Government Official** manually reviews their physical documents (Passport, Aadhar, etc.). 
- The official computes the cryptographic hash of the document.
- The official calls `verifyDocument` on the blockchain, storing this hash permanently.
- **Why?** This creates an immutable "Anchor" that serves as the gold standard for that document.

### Step 2: User-Driven Sharing
When a Bank requests a specific document, the user can:
- **Accept the Request**: The system automatically retrieves the document hash from the user's private vault.
- **Manual Copy/Paste**: The user can provide the hash directly to the Bank.

### Step 3: On-Chain Match (`checkDocumentHash`)
The Bank takes the hash provided by the user and submits it to the `checkDocumentHash` function in the smart contract.
- The contract compares: `Submitted_Hash == Anchored_Government_Hash`.
- **Match Found**: The Bank receives a cryptographic confirmation that the document the user holds is exactly the one the government verified.
- **Mismatch**: If the user tries to share a forged document or a fake hash, the blockchain will reject it immediately.

## 6. The End-to-End Workflow (The "Live" Flow)

1. **User Onboarding**: User registers with MetaMask.
2. **Gov Audit**: A Government Official manually verifies the user's identity and writes the **Document Hash** to `DecentralizedKYC.sol`.
3. **Data Encryption**: The document is encrypted and stored in the User's **ID Vault** (IPFS).
4. **Bank Request**: A Bank requests "KYC Documents".
5. **Access Grant**: User calls `grantAccess(Bank_Address)` on-chain.
6. **Hash Comparison**: The user shares their document hash. The Bank verifies this hash against the Government's "Anchor" on the blockchain.
7. **ZK-Proof Submission**: For lower-level checks (like proving age), the user submits a ZK-Proof instead of the full hash.
8. **Audit Trail**: The `Public Ledger` records that "Bank X verified User Y's Passport", but never reveals the Passport content itself.

## 7. Presentation Strategy: Key Talking Points

1. **"Selective Disclosure"**: Emphasize that we prove *attributes* of identity, not the identity itself.
2. **"Privacy by Design"**: We don't "fix" privacy later; it's built into the math of the smart contracts.
3. **"Governance through Consensus"**: Explain how MultiSig ensures no single point of failure (no rogue admin).
4. **"On-Chain vs Off-Chain"**: Be ready to explain that **Hashes** are on-chain, and **Encrypted Files** are off-chain.
